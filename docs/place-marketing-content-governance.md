# 플레이스 마케팅 콘텐츠 거버넌스 v1

본 문서는 플레이스 마케팅 지식베이스를 스레드, 블로그, 상담, 텔레그램 검수 카드에서 어떻게 사용할지 정하는 운영 기준이다. 원칙은 간단하다. `실무 실행 항목`은 내부에서 정확히 관리하고, 공개 콘텐츠는 문의를 만드는 후킹 표현으로 변환한다.

## 채널별 표현 원칙

### Threads

목표:
- 플레이스 순위/문의/광고 효율로 고민하는 운영자의 댓글, DM, 저장을 만든다.
- 기브니즈가 "내 매장 상황을 봐줄 수 있는 곳"이라는 인식을 만든다.
- 사장님의 막연한 불안을 자극만 하지 않고, 점검 가능한 플레이스 구조 문제로 바꾼다.

사용 표현:
- 플레이스 최적화
- 상위노출 구조
- 순위 개선 작업
- 유입 신호 설계
- 저장/길찾기/방문 의도 신호
- 리뷰 수보다 중요한 구조
- 광고비 전에 점검할 플레이스

사용하지 않는 표현:
- 세부 실무 실행 방식
- 순위 보장
- 자연 유입 위장
- 탐지 회피
- 대가성 표시 없는 후기 유도
- 직접 방문 여부를 우회하는 설명
- 지금 안 하면 망한다는 식의 확정적 공포 표현

### 블로그/매거진

목표:
- 플레이스 마케팅을 더 길고 신뢰도 있게 설명한다.
- 공식 안내 기준과 실무 관찰을 구분해 독자가 이해하기 쉽게 만든다.

사용 표현:
- 네이버가 공개한 작성 기준
- 고객이 비교할 때 보는 정보
- 상위노출을 위한 기본 구조
- 리뷰/사진/메뉴/설명/예약 동선 점검
- 블로그 기자단은 대가성 표시를 전제로 운영

주의:
- 스레드보다 설명이 길어도 세부 실무 실행 방식은 공개하지 않는다.
- 병의원은 의료광고 표현을 별도 검수한다.

### 상담/세일즈

목표:
- 고객의 현재 상태를 진단하고, 실행 항목을 상품 구조로 연결한다.
- 공개 콘텐츠에서 말하지 않은 실행 레버는 상담 맥락에서만 설명한다.

사용 표현:
- 현재 순위가 막힌 구간을 3단계로 진단합니다.
- SEO/리뷰/유입 신호를 나눠서 봅니다.
- 대표키워드와 첫 화면을 먼저 정리합니다.
- 이후 업종과 경쟁 강도에 맞는 실무 실행 항목을 설계합니다.
- 블로그 기자단은 대가성 표시 기준을 반영해 운영합니다.
- 순위 보장이 아니라 노출 가능성을 높이는 구조 개선입니다.

### 내부 운영

목표:
- 공개 메시지와 실제 실행을 분리해 작업자가 혼동하지 않게 한다.
- 실행량, 순서, 검수 필요 항목은 사람 담당자가 판단한다.

운영 원칙:
- 모든 실무 실행 항목은 공개 콘텐츠 자동 생성에 그대로 들어가지 않는다.
- AI는 공개 표현으로 변환한 초안만 만든다.
- 담당자는 상담 맥락에서 필요한 실행 항목을 별도로 설명한다.
- 병의원, 리뷰, 블로그 기자단, 영수증 리뷰 관련 항목은 사람이 최종 확인한다.

## 표현 변환표

| 실무 실행 항목 | 공개 후킹 표현 | 상담/세일즈 표현 | 자동화 제외 표현 |
| --- | --- | --- | --- |
| 리워드 트래픽 | 유입 신호 설계, 플레이스 활성화, 순위 개선 작업 | SEO/리뷰 기반을 맞춘 뒤 유입 신호를 설계합니다 | 탐지 회피, 자연 유입 위장, 확정 순위 |
| 영수증 리뷰 관련 운영 | 방문 고객 리뷰 동선, 리뷰 활성화 구조 | 현재 리뷰 상태와 업종에 맞게 별도 운영합니다 | 실제 방문 여부를 우회하는 설명 |
| 블로그 기자단 | 외부 검색 노출, 블로그 후기 콘텐츠, 방문 이유 설명 | 대가성 표시를 전제로 원고 방향과 키워드를 설계합니다 | 대가성 표시를 숨기는 표현 |
| 대표키워드 | 고객이 검색하는 말과 플레이스를 맞추기 | 지역/메뉴/서비스 조합을 정리합니다 | 무관한 인기 키워드 삽입 |
| 상세설명 | 첫 화면에서 선택 이유 만들기 | 키워드, 차별점, 문의 동선을 한 문단으로 정리합니다 | 근거 없는 최상급 표현 |
| 사진 | 고객이 비교할 때 필요한 사진 | 대표사진/내부/외부/메뉴/동선 사진을 재배치합니다 | 실제와 다른 이미지 추천 |
| 저장 | 다시 볼 이유 만들기 | 쿠폰/메뉴/정보 가치로 저장 이유를 설계합니다 | 보상 목적 세부 유도 방식 |
| 길찾기 | 방문 의도 신호 | 입구/주차/랜드마크/영업시간을 함께 점검합니다 | 허위 방문 의도 생성 설명 |
| 클릭/체류 | 첫 화면 이탈 줄이기 | 클릭 후 보이는 화면을 먼저 정리합니다 | 의미 없는 클릭만 늘리는 표현 |
| 병의원 후기 | 신뢰 정보와 상담 동선 | 표현 기준을 별도 확인한 뒤 작성합니다 | 치료 효과 보장, 전후 비교 과장 |

## 스레드 생성 규칙

### 공통 입력값

```json
{
  "channel": "threads",
  "target_reader": "요식업/병의원/오프라인 매장 운영자",
  "reader_problem": "플레이스 순위가 안 오르거나 광고비 대비 문의가 부족함",
  "core_frame": "순위가 안 오르는 이유는 리뷰 수 하나가 아니라 구조가 끊겨 있기 때문",
  "emotional_frame": "막연한 불안을 점검 가능한 구조 문제로 낮춰준다",
  "allowed_public_terms": [
    "플레이스 최적화",
    "상위노출 구조",
    "유입 신호 설계",
    "저장/길찾기 신호",
    "순위 개선 작업"
  ],
  "excluded_public_terms": [
    "순위 보장",
    "탐지 회피",
    "자연 유입 위장",
    "대가성 미표기",
    "실제 방문 여부 우회"
  ]
}
```

### 포맷 1. 진단형

사용 상황:
- 댓글/DM 상담을 유도하고 싶을 때
- 독자가 자기 상황을 말하게 만들고 싶을 때
- 독자의 불안을 구체적인 점검 항목으로 바꿔주고 싶을 때

구조:
1. 첫 줄: 현재 문제를 콕 집는다.
2. 둘째 줄: 기존 오해를 깬다.
3. 본문: 막힌 지점 3개를 제시한다.
4. CTA: 업종/지역/현재 키워드를 남기게 한다.

샘플:

```text
플레이스 순위가 안 오르는 사장님들,
리뷰 수만 보고 있으면 원인을 놓칠 수 있습니다.

보통 막히는 지점은 3개입니다.

1. 대표키워드가 고객 검색어와 다름
2. 첫 화면 사진/메뉴/설명이 선택 이유를 못 줌
3. 저장, 길찾기, 문의로 이어지는 유입 신호가 약함

광고비를 올리기 전에 이 3개부터 봐야 합니다.
업종이랑 지역 남겨주시면 어디부터 막혔는지 기준 잡아드릴게요.
```

### 포맷 1-1. 불안 재구성형

사용 상황:
- 광고비, 블로그, 리뷰를 계속 하고 있는데도 확신이 없는 사장님을 대상으로 할 때
- 문제를 겁주는 방식이 아니라 "어디가 막혔는지 보이게 해주는" 방식으로 풀고 싶을 때

구조:
1. 첫 줄: 사장님이 느끼는 불안을 말로 잡는다.
2. 둘째 줄: 불안의 원인을 개인 능력이 아니라 구조의 불투명함으로 바꾼다.
3. 본문: 확인할 수 있는 구조 3개를 제시한다.
4. CTA: 업종/지역/키워드를 남기면 막힌 구간을 봐준다고 닫는다.

샘플:

```text
마케팅을 하고 있는데도 계속 불안한 사장님들,
대부분 돈을 덜 써서가 아니라 어디가 막혔는지 안 보여서 그렇습니다.

플레이스에서는 보통 3곳에서 불안이 생깁니다.

1. 내 업체가 어떤 키워드로 인식되는지 모름
2. 고객이 들어와도 선택할 이유가 첫 화면에 없음
3. 저장, 길찾기, 문의로 이어지는 신호가 약함

불안하면 광고비부터 늘리게 되는데,
그 전에 구조부터 보면 낭비를 줄일 수 있습니다.

업종이랑 지역 남겨주시면
어디가 막힌 구간인지 기준 잡아드릴게요.
```

### 포맷 2. 반박형

사용 상황:
- 흔한 오해를 깨고 전문성을 보여줄 때
- "리뷰만 늘리면 된다"는 생각을 전환할 때

구조:
1. 첫 줄: 흔한 믿음을 반박한다.
2. 본문: 왜 부족한지 설명한다.
3. 실행 힌트: 먼저 볼 항목을 제시한다.
4. CTA: 현재 상태를 댓글로 남기게 한다.

샘플:

```text
리뷰 100개면 플레이스 순위가 오른다?
꼭 그렇진 않습니다.

리뷰가 있어도
대표키워드가 흐리고,
사진이 선택 이유를 못 주고,
메뉴/서비스명이 검색어와 안 맞고,
유입 신호가 약하면 순위는 잘 안 움직입니다.

리뷰는 한 조각이고,
플레이스는 구조로 봐야 합니다.

리뷰는 많은데 순위가 안 오르면
현재 잡고 싶은 키워드를 댓글로 남겨보세요.
```

### 포맷 3. 체크리스트형

사용 상황:
- 저장 가치를 만들고 싶을 때
- 사장님이 오늘 바로 점검하게 만들 때

구조:
1. 첫 줄: 저장할 이유를 준다.
2. 본문: 5개 이하 체크리스트.
3. CTA: 저장/댓글/DM.

샘플:

```text
광고비 쓰기 전에 플레이스에서 먼저 볼 5가지.

1. 대표키워드가 실제 검색어인가
2. 첫 사진이 메뉴/공간/신뢰를 보여주는가
3. 메뉴나 서비스명에 가격/설명이 있는가
4. 리뷰에 반복되는 키워드가 있는가
5. 전화, 예약, 길찾기 버튼까지 자연스럽게 이어지는가

이 5개가 비어 있으면
광고로 사람을 데려와도 전환이 잘 안 납니다.

저장해두고 오늘 플레이스 열어서 하나씩 체크해보세요.
```

### 포맷 4. 사례형

사용 상황:
- 상담 전환을 자연스럽게 만들 때
- 추상적 설명보다 "이런 순서로 한다"를 보여줄 때

구조:
1. 첫 줄: A 매장의 문제.
2. 중간: 바꾼 순서.
3. 끝: 독자 상황 적용 질문.

샘플:

```text
어떤 식당은 블로그를 계속 써도
플레이스 순위가 거의 안 움직였습니다.

봤더니 문제는 글 수가 아니었습니다.

대표키워드는 너무 넓고,
첫 사진은 메뉴보다 인테리어 위주였고,
메뉴 설명에는 고객이 검색하는 단어가 빠져 있었습니다.

그래서 순서를 바꿨습니다.

1. 대표키워드 재정리
2. 대표사진 교체
3. 메뉴명/설명 정리
4. 리뷰에서 반복될 경험 키워드 정리
5. 이후 유입 신호 설계

플레이스는 하나만 고쳐서 되는 경우보다
막힌 순서를 찾아야 움직이는 경우가 많습니다.
```

## 텔레그램 검수 카드 기준

향후 스레드 초안 검수 카드에는 아래 항목을 표시한다.

```json
{
  "channel": "threads",
  "format_type": "diagnostic | rebuttal | checklist | case",
  "target_reader": "",
  "reader_problem": "",
  "public_hook": "",
  "internal_basis": "",
  "editable_public_terms": [],
  "blocked_terms": [],
  "recommended_cta": "",
  "review_note": ""
}
```

검수자가 확인할 질문:

- 이 글은 누구에게 말하는지 첫 2줄 안에 보이는가?
- 문제와 읽을 이유가 충분히 직접적인가?
- 실무 실행 항목명이 공개 콘텐츠에 직접 노출되지 않았는가?
- 댓글/DM으로 이어지는 이유가 자연스러운가?
- 병의원 콘텐츠라면 별도 검수 필요 문구가 있는가?

## 문서 검수 기준

- 민감한 실무 분류에는 `실무 실행 항목`이라는 명칭을 사용한다.
- `실무 실행 항목`은 공개 콘텐츠 문장으로 자동 삽입하지 않는다.
- 공개 후킹 표현과 내부 세일즈 표현을 분리한다.
- 블로그 기자단은 대가성 표시 전제를 유지한다.
- 자동화 제외 표현은 생성 프롬프트의 금지어/검수어로 사용한다.
